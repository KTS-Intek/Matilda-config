  1. Packets

Packet format

Text JSON (ECMA-404 standard). UTF-8 text codec. Data is always transmitted in text mode, if necessary, encoded in base64 (data can be transmitted with an alignment to a ratio of 3:4 or without, an alignment is allowed with the symbol ‘=’).

{"cmd":"0 < x < 60000", "key1":"<some data>", "key2":"<value1>",...., "Md5":"<hash sum>"}

There are two types of keys: global and query-dependent.

Reserved keys: Md4, Md5, Sha1, Sha224, Sha256, Sha384, Sha512, Sha3_224, Sha3_256, Sha3_384, Sha3_512, cmd, cmprss

If a packet is compressed, the hash is calculated for both the packet to be compressed and the packet in which the compressed data will be transmitted.

Hash

 Md5 (details described below) is used by default.

Compression

Compression permission is granted upon authorization. If enabled, up to 500 bytes of the packet is transmitted as is, otherwise the packet is prepared as usual, compressed by the zlib algorithm (or other available) and encoded in base64, then a packet is created with the key "cmd" which corresponds to compressed data and key name compression algorithm which corresponds to the compressed data encoded in base64:

 {"cmd":"8", "zlib":" <toBase64( <(uint32)first packet len> zlib_compress(<first packet>,9))>", “Md5”:”<hash summ>”}

Since in networks with low latency and high bandwidth, the use of high data compression is not always appropriate, to optimize the data rate, the server measures network delays and adjusts the packet size, and manages the need for compression.

Response size is allowed for some commands.

Hash calculation sequence

Example A

  1. Packet: {"cmd":1,"value":"qwerty","Md5":"0"};
  2. Hash Md5, A5 2D 64 09 C7 FB 1B 64 7B 71 43 41 5D D7 19 41  or in base64 pS1kCcf7G2R7cUNBXdcZQQ.
  3. Replacing the hash value {"cmd":1,"value":"qwerty","Md5":"pS1kCcf7G2R7cUNBXdcZQQ"}

Example B

  1. Packet: {"cmd":1,"value":"йцукен","Md5":"0"};
  2. UTF-8 encoding:  7b 22 63 6d 64 22 3a 31 2c 22 76 61 6c 75 65 22 3a 22 d0 b9 d1 86 d1 83 d0 ba d0 b5 d0 bd 22 2c 22 4d 64 35 22 3a 22 30 22 7d;
  3. Hash Md5:  80 d2 ec 59 6b 56 9f 7f cf 87 78 36 e9 2c dd ad, або в base64: gNLsWWtWn3/Ph3g26SzdrQ.
  4. Replacing the hash value: {"cmd":1,"value":"йцукен","Md5":"gNLsWWtWn3/Ph3g26SzdrQ"};

Forming a request, getting an answer

A typical scheme of the packet: <key>:<value>

{"cmd":"0 < x < 65000", "key1":"<some data>", "key2":"<value1>",....,"Md5":"toBase64(<hash summ>)"}

The sequence of preparation of the packet to calculate the hash sum:

  1. All keys and data are added to the packet

{"cmd":"0 < x < 65000", "key1":"<some data>", "key2":"<value1>"}

  1. A key with the name of the hash sum is added to the packet, “0”is used as data

{"cmd":"0 < x < 65000", "key1":"<some data>", "key2":"<value1>",....,"Md5":"0"}   //Md5 zero is passed as a text field

  1. The hash of the received packet is inserted into the corresponding key

{"cmd":"0 < x < 65000", "key1":"<some data>", "key2":"<value1>",....,"Md5":"toBase64(<Md5 hash summ>)"}

  1. M2M service connections

A UC is constantly connected to the specified servers (service channel). To connect to the device, it is necessary to specify one of its MAC addresses or Object ID.

There may be the following situations:

  • The specified device is connected to the service. If the specified device is connected to the service, a temporary port will be allocated for the incoming connection (data channel) through which the device will be exchanged. Data is transmitted in transparent mode without buffering via the data channel, unless data is transmitted from one incoming connection and the second connection has not yet been activated (the client has not connected), immediately after connecting the second client all the contents of the buffer will be transferred. Authorization and subsequent exchange is performed similarly to a direct connection. When one of the clients is disconnected, the data channel service is closed.

Example

Request: search for a device with ID "matilda2".

{"cmd":11,"cmmprssn":"zlib","remote":"matilda2","useId":true,"version":1, "Md5":"C2agCeXEXuaEj3Evsj9zPA"}

Answer: The device is found. Connect via kts.ddns.net,  port is 50000.

{"cmd":11,"sIp":"kts.ddns.net","sP":50000, "Md5":"Bx8nUJW2m3OlHtQgbz7Lpg=="}

  • Many devices with the same ID are connected to the service (connection was by ID). A list consisting of <MAC>@<Object ID> pairs will be sent. The MAC address is the key value (when connected by ID, the associated MAC address is searched) and unique (when connecting devices with the same MAC, the one that was registered last will be active). To continue the connection, you must select the required MAC address and make the connection using the MAC.

Example

Request: search for a device with ID "matilda1" (2 devices with similar ID are connected to the server)

{"cmd":11,"cmmprssn":"zlib","remote":"matilda1","useId":true,"version":1, "Md5":"NGXIVtKcLGNsU/p9lKOd1w"}

Answer:  found several devices with the same ID “matilda1”

{"cmd":11,"l":["6C:6C:6С:6С:6С:6С@matilda1","6E:6E:6E:6E:6E:6E@matilda1","6D:6D:6D:6D:6D:6D@matilda1","6F:6F:6F:6F:6F:6F@matilda1"], "Md5":"SolUU7Me+iZCmve80d+jOw=="}

Request: connection to the device with MAC "6C:6C:6С:6С:6С:6С" (on the server all connected devices are separated by MAC address)

{"cmd":11,"cmmprssn":"zlib","remote":"6C:6C:6С:6С:6С:6С","useId":false,"version":1, "Md5":"CLduS1GdWmU69J3r9F/7Jw=="}

Answer: The device is found. Connect via kts.ddns.net,  port is 50000.

{"cmd":11,"sIp":"kts.ddns.net","sP":50000, "Md5":"Bx8nUJW2m3OlHtQgbz7Lpg=="}

  • The specified device is not connected to the service.

Example

Request: search for a device with ID "matilda22"

{"cmd":11,"cmmprssn":"zlib","remote":"matilda22","useId":true,"version":1, "Md5":"z+fCyfAC1/FdgRDxBznVVA"}

Answer: the device is not found

{"cmd":13,"e":12,"em":"Unknown device","lcmd":11, "Md5":"kJ7/tTRPfWNhCFLGRcOlcw=="}

  1. Appendix G

Protocol V1 comment field

In the general case, the comment to the event is transmitted in the comment column, in case it is an event from the meter, the data is transmitted in the following format:

0xCC opt<Text comment>, where

0xCC — meter event code;

opt — event clarification (used only in some versions of meters), separated from 0xCC by a space, if it is not used, it is passed empty, and the space in front is deleted;

<Text comment> — textual explanation of the event.

Depending on the meter type, 0xCC can be transmitted as a number or as a text field (e.g. CE303).

Meter readings, cell merge rules for jns key

Protocol V2 and higher for the  #COMMAND_READ_DATABASE command

Key jns describes in what form and/or sequence the data will be merged. At the moment, the most "economical" key is 6.

0

default key, a regular data processing, do not change anything

1

merge cells with data statuses, to save traffic, it is relevant if a large number of meters are configured to poll not for all energies and/or not for all tariffs

2

 it is similar to 1, but this indicates that the meter_ni column should be omitted too

3

complements 1 by changing the data sequence for profiles containing tariffs. For example, if the request had a sequence of energies A +, A-, R +, R- and all tariffs, then the answer will be the format Tx_A +, Tx_A-, Tx_R +, Tx_R-, where x

will always be from 0 to 4 inclusive.

4

complements 2 by changing the data sequence for profiles containing tariffs. For example, if the request had a sequence of energies A +, A-, R +, R- and all tariffs, then the answer will be the format Tx_A +, Tx_A-, Tx_R +, Tx_R-, where x

will always be from 0 to 4 inclusive.

5

complements 3, so that the values in the array corresponding to the values of the meter are combined into one TS by symbol "|".

6

complements 4, so that the values in the array corresponding to the values of the meter are combined into one TS by symbol "|".

Meter events, cell merge rules for the smpl key

Protocol V2 and higher for the  #COMMAND_READ_METER_LOGS command

Key smpl describes which columns will be transmitted. At the moment, the "most economical" is the value of 5.

0

date_time — date and time in UTC time zone;

ucm_event_code — unique event code (see Appendix I)

meter_model — the name of the plugin with which the meter was polled

meter_ev_code — meter event code (depends on the meter model).

Power profile line

The format is <time>-<power in percents>, where

time 

- fixed minutes from the beginning of the day

- a time variable - $sunrise or $sunset, it is possible to add or subtract up to 180 minutes. The device coordinate and time zone are used to calculate these variables.

power in percent - 0 - turn off, 1...100 turn on, 100 the full power.

Example

The line ‘00:00-60,04:00-80,$sunrise-10-35,$sunrise-5-0,$sunset-10-35,$sunset+3-80,22:00-70’ means

00:00 - 60 [%]

04:00 - 80 [%]

$sunrise-10 - 35 - ten minutes before sunrise, 35 [%]

$sunrise-5 - 0 - five minutes before sunrise, turn off

$sunset-10 - 35 - ten minutes before sunset, turn on at 35  [%]

$sunset+3 - three minutes after sunset,  80  [%]

22:00 - 70  [%]

Special profile line

A UTC time zone is used to determine when to apply a schedule from special schedules.

The format is

YYYYMMDDHHMMWmmmm, where

YYYY - year

MM - month

DD - day of the month

HH - hour, 24 hour format

MM - minutes

W - day of the week, 0 - Sunday, 1 - Monday

mmmm - duration in minutes

YYYYMMDDHHMMW is used to know when to apply, mmmm is used to limit the duration of the schedule

character ‘*’ means any value;

The line ‘****12311151*2880’

**** - any year

12 - December

31 - day of the month

11 - hour

51 - minute

* - any day of the week

2880 - the duration is 2 days
